## 역할 및 프로필

- 1인 풀스택 + 팀 리드: 기획, 디자인, 프론트/백엔드, DB, 배포까지 전 과정 직접 수행
- frontA/B/C 브랜치 팀원들의 작업을 통합하고 최종 품질 관리
- AI 도구(Claude)를 적극 활용하여 빠른 프로토타이핑 + 반복 개선

## 의사결정 패턴

### 1. UI/UX 결정: "샘플 비교 → 즉시 판단"
- 항상 2~3개 시각적 샘플을 요청한 뒤 비교하여 선택
- 예: 시민 아이콘 3종 비교 → 2번 선택, GPT 테두리 inset vs gradient → inset 선택
- **근거**: 말로 설명하기보다 눈으로 직접 확인해야 확신이 생김
- **Claude 피드백**: 이 방식은 매우 효율적. 다만 샘플 제시 시 각 옵션의 트레이드오프를 미리 설명하면 선택 속도가 더 빨라질 수 있음

### 2. 실패 판단이 빠름: "안 되면 바로 롤백"
- 시도 후 1~2번 내에 맞지 않으면 즉시 원복 지시
- 예: 스코어보드 라이트 테마 → "별론데" → fight3 배경 → "안울린다" → 네이비 원복
- 예: 레이더 차트 숫자 → 여러 시도 후 "그냥 원래대로 지워"
- **근거**: 완벽하지 않은 것에 시간 쓰기보다 작동하는 상태를 유지하는 것이 낫다는 판단
- **Claude 피드백**: 좋은 판단력. 다만 기술적 한계(Canvas 렌더링 등)로 실패한 건 나중에 다른 접근(HTML 오버레이 등)으로 재시도할 가치가 있을 수 있음

### 3. 디테일 집착: "1px 단위까지 조정"
- 아바타 테두리 두께, 글로우 강도, 투명도 수치까지 세밀하게 지시
- 예: "흰색0.5 검은색2 흰색0.5", "opacity 30%→100% 조건부"
- **근거**: 모바일 퍼스트 440px 화면에서 1px 차이가 체감됨
- **Claude 피드백**: 모바일에서의 디테일 감각이 뛰어남. CSS 구현 한계(img에 inset shadow 불가 등)를 미리 공유하면 불필요한 시도를 줄일 수 있음

### 4. 용어/일관성 강조
- "A/B → 찬성/반대", "GPT→Gemini→Claude 순서 고정" 등 전체 통일 지시
- **근거**: 사용자 경험의 일관성이 서비스 신뢰도를 높인다는 확신
- **Claude 피드백**: 매우 적절한 판단. 공통 상수(judges.js) 추출로 이어진 좋은 사례

### 5. 기능보다 느낌 우선
- "어울리나?", "느낌 보려고" 등 감성적 판단이 먼저
- 기술적 완성도보다 전체적인 톤앤매너 조화를 중시
- **근거**: 법정/아고라 메타포라는 서비스 정체성에 맞아야 함
- **Claude 피드백**: 브랜딩 감각이 명확함. 새 기능 추가 시 "이 메타포에 어울리는가?"를 체크리스트로 쓸 수 있음

## 작업 지시 패턴

### 잘하는 부분
- **명확한 시각적 피드백**: 스크린샷 + 구체적 지적 ("여기 색상이 반대", "이게 더 커보여")
- **스코프 조절**: 과도한 요청 시 스스로 "그냥 원래대로" 판단
- **병렬 작업 관리**: 여러 파일/기능을 동시에 개선하면서 일관성 유지

### 보완하면 좋을 부분
- **요구사항 선행 정리**: 때로 "이것도 해봐" → "아니 저것도" → "원래대로" 루프 발생. 변경 전에 "목표 상태"를 한 문장으로 정리하면 반복 줄일 수 있음
- **기술적 제약 사전 확인**: CSS/Canvas 한계를 모른 채 시도 → 실패 → 롤백 패턴이 간혹 발생. "이거 가능해?"를 먼저 물어보면 시간 절약
- **테스트 데이터 다양화**: 현재 반대측이 항상 이기는 데이터만 있어서 찬성 승리 UI를 확인 못함. 양쪽 케이스 테스트 데이터 준비 필요

## Claude 종합 피드백

**강점**: 빠른 판단력, 시각적 감각, 일관성에 대한 높은 기준, 실패 시 신속한 방향 전환
**제안**:
1. 새 UI 작업 시작 전 "변경 목표 + 제약조건"을 한 줄로 정리하면 시행착오 50% 감소 예상
2. 테스트 시나리오(찬성 승/반대 승/무승부/투표 미달)별 확인 체크리스트 도입 권장
3. 배포 전 주요 플로우 E2E 테스트 시나리오 정리하면 프로덕션 버그 예방에 도움
