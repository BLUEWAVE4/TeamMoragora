1 요구사항 정의서

개요
Moragora 전체 기능 범위에 맞춰 데이터베이스 요구사항을 정의합니다.
총 17개 테이블, Supabase Auth 연동, JSONB 기반 채점 상세, RLS(Row Level Security) 정책 적용, 리그 시스템(XP/티어), 실시간 채팅, 알림, 좋아요, 별점, 분석 이벤트 지원.
번호 요구사항 ID 요구사항명 요구사항 상세 관련 객체 비고
1 DB-REQ-001 사용자 프로필 관리 Supabase Auth(auth.users)와 연동하여 사용자 프로필을 생성·관리한다. 닉네임, 프로필 이미지, 승/패/무 전적, 성별, 나이, 누적 점수를 저장하고 마이페이지에서 조회한다. profiles Auth 연동 PK
2 DB-REQ-002 논쟁 방 생성 및 관리 사용자가 논쟁 주제, 카테고리(7종), 판결 목적(compete/consensus/analyze), 분석 렌즈(6종)를 설정하여 논쟁 방을 생성한다. 초대 코드(UNIQUE)로 상대방을 초대하며, 상태(waiting→both_joined→chatting→arguing→judging→voting→completed) 전이를 관리한다. 논쟁 모드(duo/solo/chat/daily), 채팅 마감시각, 조회수를 관리한다. debates 상태 머신 관리
3 DB-REQ-003 양측 주장 저장 논쟁 참여자 A/B 양측이 50~2,000자 범위의 주장을 작성한다. 논쟁 ID, 작성자 ID, 진영(A/B)을 기록하여 AI 판결 시 입력 데이터로 활용한다. arguments 콘텐츠 필터링 대상
4 DB-REQ-004 AI 종합 판결 저장 3개 AI(GPT-4.1, Gemini 2.5 Flash, Claude Sonnet)의 개별 점수를 평균하여 AI 종합 점수를 산출하고, 시민 투표 점수와 복합(AI 각 25% + 시민 25%)하여 최종 판결을 저장한다. 논쟁당 1건(1:1 관계), 시민 투표 30명 이상 시 반영 플래그 관리. AI 판결은 즉시 표시하고, 투표 마감(vote_deadline) 후 시민 점수를 합산하여 최종 판결을 확정한다. verdicts 1:1 관계 (debates)
5 DB-REQ-005 AI 개별 판결 기록 GPT-4.1, Gemini 2.5 Flash, Claude Sonnet 3사 각각의 판결문, 승패 판정, 총점(0~100), 5개 항목별 채점 상세(JSONB: logic/evidence/rebuttal/consistency/expression × 0~20), 확신도(0.00~1.00), 처리 상태(success/failed/timeout)를 저장한다. ai_judgments JSONB 채점 상세
6 DB-REQ-006 시민 투표 관리 로그인 사용자가 완료된 논쟁에 대해 A 또는 B 측에 투표한다. 논쟁당 1인 1투표(UNIQUE 복합 제약)를 보장하며, 30명 이상 투표 시 시민 점수를 최종 판결에 반영한다. votes UNIQUE(debate_id, user_id)
7 DB-REQ-007 콘텐츠 필터링 로그 3단계 콘텐츠 방어(1단계: 비속어 사전, 2단계: AI 필터(Gemini Flash), 3단계: 게이트키핑) 처리 결과(pass/warn/block)와 사유를 기록한다. 주장(argument)과 주제(topic) 모두 필터링 대상. content_filter_logs 3단계 필터링
8 DB-REQ-008 리그 시스템 (XP/티어) 사용자의 활동(승리, 패배, 투표 등)에 따라 XP를 부여하고 이력을 기록한다. XP 총합에 따라 5단계 티어(시민→배심원→변호사→판사→대법관)가 결정되며, 기간별(월별) 집계를 지원한다. profiles, xp_logs 리그/랭킹 기능
9 DB-REQ-009 댓글 시스템 판결 결과에 대해 사용자가 댓글을 작성하고, 다른 사용자의 댓글에 좋아요를 할 수 있다. 1인 1좋아요(UNIQUE 복합 제약)를 보장하며, likes_count 비정규화 카운트를 관리한다. comments, comment_likes TIER 2
10 DB-REQ-010 Row Level Security 모든 테이블에 RLS를 적용하여 auth.uid() 기반 데이터 접근 제어를 수행한다. 프로필은 전체 조회/본인 수정, 판결·AI판결은 서버만 생성, 투표는 로그인 사용자 생성/본인 삭제. 전체 테이블 Supabase RLS
11 DB-REQ-011 입력 전처리 주장 제출 시 preprocessor.service.js로 프롬프트 인젝션 방어(15개 패턴), HTML/마크다운 제거, 유사도 검사(Jaccard >85% 거부)를 수행한 후 arguments 테이블에 삽입한다. 최소 50자, 최대 2000자 제한. arguments 입력 새니타이징
12 DB-REQ-012 프로필 전적 자동 갱신 판결 확정(finalizeVerdict) 시 creator=A측, opponent=B측으로 매핑하여 wins/losses/draws를 +1 처리하고, total_score에 해당 논쟁 최종 점수를 누적 합산한다. profiles verdict 확정 트리거
13 DB-REQ-013 솔로 모드 mode='solo' 논쟁은 생성 즉시 status='arguing'으로 시작한다. 사용자가 A측 주장 제출 후, GPT-4o가 B측 반대 주장을 자동 생성하여 arguments에 저장한다. opponent_id는 NULL. debates, arguments AI 반대 주장 생성
14 DB-REQ-014 사용자 피드백 로그인 사용자가 서비스 만족도 피드백을 제출한다. 5항목(만족도, AI정확도, UI편의성, 공정성, 추천의향) 0.5~5.0점 평점 + 텍스트 피드백 3개 필드. 1인 1피드백(user_id UNIQUE), upsert 방식으로 기존 피드백 수정 가능. feedbacks UNIQUE(user_id)
15 DB-REQ-015 실시간 채팅 메시지 논쟁 참여자가 실시간 채팅으로 주고받은 메시지를 저장한다. 논쟁 ID, 작성자 ID, 닉네임, 내용, 진영(A/B), 작성 시각을 기록하여 채팅 로그를 보존하고, 이후 AI 판결 입력 데이터로 활용할 수 있다. chat_messages Socket.IO 연동
16 DB-REQ-016 알림 저장 사용자에게 발송되는 알림(채팅 시작, 판결 완료, 투표 리마인더 등)을 저장한다. 알림 유형, 제목, 메시지, 링크, 읽음 여부를 관리하여 알림 목록 조회 및 미읽음 카운트를 지원한다. notifications 알림 시스템
17 DB-REQ-017 논쟁 좋아요 로그인 사용자가 관심 있는 논쟁에 좋아요를 할 수 있다. 논쟁당 1인 1좋아요(UNIQUE 복합 제약: debate_id + user_id)를 보장하며, 인기 논쟁 랭킹에 활용한다. debate_likes UNIQUE(debate_id, user_id)
18 DB-REQ-018 판결 별점 사용자가 판결 결과에 대해 0.5~5.0점 범위의 별점을 매길 수 있다. 판결당 1인 1별점(UNIQUE 복합 제약: verdict_id + user_id)을 보장하며, AI 판결 품질 피드백에 활용한다. verdict_ratings UNIQUE(verdict_id, user_id)
19 DB-REQ-019 페이지 조회 분석 사용자의 페이지 조회 이력을 저장한다. 경로, 세션 ID, 사용자 ID(로그인 시), 유입 경로(referrer), 브라우저 정보(user_agent)를 기록하여 트래픽 분석 및 사용자 행동 추적에 활용한다. page_views 분석/트래픽
20 DB-REQ-020 커스텀 이벤트 추적 서비스 내 주요 이벤트(논쟁 생성, 판결 조회, 투표 참여 등)를 커스텀 이벤트로 기록한다. 이벤트명, 메타데이터(JSONB), 세션 ID, 사용자 ID, 경로를 저장하여 서비스 분석 대시보드에 활용한다. analytics_events JSONB 메타데이터
